# This Tech-CEO's Claude Code Toolkit Will Blow You Away

## Summary

Sean Kochel walks through five standout skills from the Y Combinator CEO's public Claude Code skills library — a collection of nearly 30 reusable commands designed to help builders avoid the classic mistake of shipping too much of the wrong thing. The video frames the library through a painful personal lesson: Kochel spent two years and a million dollars building features nobody wanted. The YC toolkit is positioned as a systematic corrective, bringing startup-grade product discipline (demand validation, wedge thinking, competitor research) directly into the Claude Code workflow.

The bulk of the video is a live walkthrough of the "Office Hours" skill, where Kochel uses it to pressure-test a real app idea — an agent-native nutrition tracking app meant to replace the aging Cronometer with an LLM reasoning layer. The skill walks him through a structured interview: evidence of demand, failure modes, target user, existing workarounds, minimal wedge, and long-term defensibility. It then autonomously runs web searches to map the competitive landscape before producing wireframes and a design document. The interaction demonstrates how the skill refuses surface-level answers and pushes for honest product thinking.

The video also briefly introduces a "CEO Review" skill, which is designed to uncover "10-star concepts" hiding within your current product idea — the logic being that the most valuable thing you're building is often not the thing you think you're building (the Slack-from-a-gaming-company example). Together, these skills represent a pattern of encoding YC-style mentorship directly into the agentic development loop, turning Claude Code into a disciplined co-founder rather than just a fast code generator.

## Key Concepts

### YC Skills Library as Structured Product Discipline
The Y Combinator CEO has published a library of ~30 Claude Code skills that inject startup methodology into the vibe-coding workflow. Rather than just accelerating code output, these skills enforce thinking about demand validation, wedge features, and defensibility *before* building begins. This is a direct response to the pattern of fast builders shipping large amounts of unwanted features — "vibe coding mistakes just done very fast."

### Office Hours Skill: Demand Interrogation Flow
The Office Hours skill runs a scripted, branching interview that mirrors real YC partner sessions. It asks for: (1) project type, (2) strongest evidence of real demand ("would users be upset if it disappeared?"), (3) prior failure modes, (4) target user profile, (5) current workarounds/competitors, (6) the smallest payable wedge, and (7) long-term defensibility risk. After the interview, it autonomously executes web searches across competitive spaces, surfacing market conversations and open-source alternatives before synthesizing core premises and wireframes.

### Wedge-First Thinking
A recurring emphasis throughout the skill interaction is identifying the *minimum thing someone would actually pay for on day one* — not the full vision. In the nutrition app example, the skill correctly identifies "import existing data + natural language query" as a more honest wedge than "full AI food logging." This forces a discipline that prevents over-building and gives a faster signal on whether demand is real.

### Multi-Model Validation
The Office Hours skill offers an optional step to use a secondary model (Codex is mentioned) to challenge the premises generated by Claude. This is a lightweight implementation of the builder/validator pattern — one model generates premises, another stress-tests them — applied at the product strategy layer rather than the code layer.

### CEO Review Skill: Finding the Hidden 10-Star Product
The CEO Review skill analyzes existing product docs to identify whether a more valuable concept is embedded within what you're building. The Slack origin story is the illustrative case: the product that became a billion-dollar company was a throwaway internal tool inside a failing game studio. The skill is designed to surface these overlooked pivots before builders have gone too far down the wrong path.

## Practical Takeaways

- **Run the Office Hours skill before writing a single line of code.** The structured demand interrogation will surface whether your wedge is real and expose the competitive risk (e.g., "could Cronometer just add this?") before you've sunk time into building.
- **Don't accept the skill's premises uncritically.** Kochel pushes back on the "logging is secondary to querying" premise because his use case genuinely requires both. The skill is a starting point for reasoning, not a verdict.
- **Use the wedge question as a forcing function.** Ask: "What is the smallest version someone would pay for *this week*?" If the answer requires six months of development, the wedge isn't real yet.
- **Pair the Office Hours output with the Engineering Review and CEO Review skills** before committing to build. The skills are designed to be run in sequence, with each one pressure-testing a different layer: demand → strategy → technical feasibility.
- **The skills library pattern is replicable.** The value is less about these specific skills and more about the idea of encoding expert interview frameworks into Claude Code commands. Teams can build their own domain-specific equivalents.
